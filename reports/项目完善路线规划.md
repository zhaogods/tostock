# 项目完善路线规划

> 生成日期：2026-06-01
>
> 范围：基于当前仓库代码、README、CLAUDE.md 和关键入口文件的静态梳理。

## 1. 项目定位

tostock 是一个面向 A 股的本地化量化分析工作台，核心能力覆盖：

- 行情与基础数据抓取
- 技术指标计算
- K 线形态识别
- 规则策略选股
- 回测验证
- Web 可视化展示
- Docker / Supervisor 部署

它更适合个人投资者、小团队和本地研究环境，而不是复杂的机构级交易平台。

## 2. 当前完整流程

### 2.1 配置与启动

项目主要通过以下层级读取配置：

1. 环境变量
2. 根目录 `.env`
3. `instock/config/*.json`
4. 代码默认值

关键配置包括：

- 数据库连接
- Tushare Token 与频控参数
- 东方财富 Cookie
- 代理配置
- 交易客户端配置

### 2.2 盘中轻量流程

盘中任务入口是 `instock/job/realtime_only_job.py`，主要用于：

- 初始化数据库
- 刷新个股与 ETF 实时行情

这个流程适合在交易时段每 30 分钟执行一次，减少对外部接口的压力。

### 2.3 收盘后全量流程

主入口是 `instock/job/execute_daily_job.py`，当前流程大致为：

1. 初始化数据库
2. 抓取基础行情
3. 抓取综合选股数据
4. 抓取收盘后数据
5. 并行执行：
   - 其他基础数据
   - 技术指标计算
   - K 线形态识别
   - 策略选股
6. 回测脚本保留，但不在主流程中自动启用

### 2.4 数据分层

`instock/core/stockfetch.py` 负责统一调度数据源，整体是三层结构：

- Tushare：个股行情、资金流、历史行情
- AkShare：ETF 行情、事件类数据、交易日历
- 东方财富 legacy crawler：选股、板块资金流、抢筹、涨停原因

### 2.5 Web 展示

Web 是 Tornado 服务端渲染模式，不是前后端分离 SPA。

核心流程：

- `instock/web/web_service.py` 负责路由与启动
- `instock/core/singleton_stock_web_module_data.py` 根据 `tablestructure.py` 生成菜单与页面定义
- `instock/web/dataTableHandler.py` 提供通用表格页
- `instock/web/dataIndicatorsHandler.py` 提供指标与图表页

### 2.6 Docker 部署

Docker 镜像通过 `docker/Dockerfile` 构建，容器内由 Supervisor 启动：

- 日常任务
- Web 服务
- Cron 调度

同时配合 `docker/build.sh` 生成和推送镜像。

## 3. 这个项目现在能做什么

### 3.1 面向个人投资者

可以作为一个本地 A 股研究台，支持：

- 自动采集每日行情
- 查看技术指标买卖信号
- 查看 K 线形态信号
- 跑规则型策略选股
- 查看 Web 图表与明细
- 定时更新观察池

### 3.2 面向量化研究

可以作为轻量策略研究框架，支持：

- 日频数据沉淀
- 指标研究
- 规则策略扫描
- 选股结果验证
- 历史缓存复用

### 3.3 可扩展方向

未来可以进一步演化成：

- 策略研究平台
- 因子分析平台
- AI 复盘助手
- 股票观察池与预警系统

## 4. 当前主要短板

### 4.1 工程与安全

- SQL 更新逻辑存在字符串拼接风险
- Web 生产配置不够硬化
- Supervisor 管理端口有暴露风险
- 缺少测试、lint、类型检查
- GitHub Actions 当前不适合作为可信 CI

### 4.2 数据可靠性

- 多数据源之间缺少系统性校验
- 抓取失败常表现为静默返回空结果
- 历史缓存使用 pickle，缺少 schema 版本与损坏检测
- 任务编排依赖固定 sleep，不够显式

### 4.3 策略与回测

- 策略偏规则筛选，缺少统一策略接口
- 回测没有并入主流程
- 回测指标较少，缺少资金曲线、最大回撤、夏普等核心统计
- 没有交易成本、滑点和仓位管理

### 4.4 产品化能力

- 缺少任务运行日志总览
- 缺少告警与推送
- 缺少结构化 AI 复盘输出
- 缺少 API 化与前后端分离能力

## 5. 完善路线图

## P0：先把项目跑稳

目标：提升可用性、可部署性和基础安全性。

建议优先做：

1. 统一配置加载逻辑
   - 把 `.env`、数据库配置、Tushare 配置收拢到单一配置入口
2. 修复 Web 生产配置
   - 关闭 debug
   - cookie secret 强制外置
   - 增加 health 接口
3. 修复 SQL 拼接风险
   - 改为参数化 SQL
4. 收紧 Supervisor 暴露面
   - 管理端口只监听本地或加认证
5. 修复或移除无效 CI
   - 让 GitHub Actions 可信可用

### 预期收益

- 部署更稳
- 出错更容易定位
- 安全面更可控

## P1：增强数据质量

目标：让数据可追踪、可验证、可恢复。

建议优先做：

1. 增加 job 运行日志表
   - 记录每次任务的成功、失败、写入量、耗时
2. 增加数据质量校验
   - 日期合法性
   - 行数异常检测
   - 空值比例检测
   - 价格和成交量边界检查
3. 增加多源交叉验证
   - 对关键字段做一致性检查
4. 升级历史缓存格式
   - 从 pickle 迁移到 Parquet
   - 增加 schema 版本与校验信息
5. 增加失败告警
   - 任务失败后提醒，而不是只写日志

### 预期收益

- 数据更可信
- 策略结果更稳定
- 便于长期运行

## P2：把回测做强

目标：让策略不只是“选出来看看”，而是“有统计证据”。

建议优先做：

1. 启用并重构回测流程
   - 保留手动回测入口
   - 可按配置决定是否进主流程
2. 扩展标准回测指标
   - 累计收益
   - 年化收益
   - 最大回撤
   - 夏普比率
   - 胜率
   - 盈亏比
3. 增加组合级回测
   - 等权买入
   - 调仓逻辑
   - 资金曲线
4. 加入交易成本建模
   - 手续费
   - 印花税
   - 滑点
   - 涨跌停成交约束
5. 形成策略对比体系
   - 不同策略的横向对比
   - 命中率与收益率排行

### 预期收益

- 回测更接近真实交易
- 策略优劣更容易量化比较
- 项目从“筛选工具”升级为“研究工具”

## P3：策略平台化

目标：从写死脚本变成可配置、可扩展的策略系统。

建议优先做：

1. 统一策略接口
   - 统一准备数据、生成信号、输出结果的约定
2. 策略参数化
   - 把阈值改为配置项
3. 策略结果标准化
   - 统一命中结果表结构
4. 策略排行榜
   - 按近期表现、胜率、回撤、收益排序
5. 策略调参支持
   - 支持批量参数组合评估

### 预期收益

- 新策略接入更简单
- 更适合持续迭代
- 便于后续做因子研究

## P4：做成 AI 决策助手

目标：把已有结构化数据转化为自然语言洞察。

建议优先做：

1. 每日市场复盘
   - 资金流
   - 板块轮动
   - 涨停原因
   - 市场风格
2. 个股分析报告
   - 趋势
   - 指标
   - 资金流
   - 事件面
   - 风险点
3. 策略命中解释
   - 为什么命中
   - 历史类似样本如何
   - 当前风险在哪里
4. 观察池生成
   - 强势池
   - 超跌反弹池
   - 突破池
   - 风险名单
5. 通知推送
   - 飞书
   - 邮件
   - Telegram
   - 微信机器人

### 预期收益

- 项目价值从“数据展示”提升到“决策辅助”
- 更贴合“量化 + AI”定位

## 6. 推荐落地顺序

如果要按性价比排序，建议先做下面 5 件事：

1. 统一配置与 `.env` 加载
2. 做任务运行日志
3. 修复 SQL 拼接和 Web 生产配置
4. 增强回测指标与组合回测
5. 做每日 AI 复盘报告

这个顺序能同时改善：

- 安全性
- 稳定性
- 数据可信度
- 策略研究能力
- 产品表现力

## 7. 最终目标形态

最合理的终态不是自动交易机器人，而是：

> A 股个人量化研究 + 数据看板 + AI 复盘助手

理想的数据流可以是：

```text
每日自动更新数据
    ↓
生成行情 / 资金 / 指标 / 策略结果
    ↓
自动回测验证策略质量
    ↓
Web 展示策略排行榜和股票详情
    ↓
AI 生成每日复盘和个股分析
    ↓
推送观察池和风险提醒
```

这个方向既能保留当前项目的工程资产，也能逐步补齐产品、研究和 AI 能力。
