## 一、概览

该仓库是一个以 `instock/` 为核心的 Python 股票数据系统，主要包含以下能力：

- 行情与数据抓取
- 日常批处理作业
- Tornado Web 可视化界面
- 基于 `easytrader` 的可选自动交易能力
- Docker + cron + supervisor 的容器化运行方式

已完成的初始化产物：

- 仓库级操作说明：`CLAUDE.md`

## 三、代码架构总结

### 3.1 主体结构

代码主包为 `instock/`，大致分为四层：

- `instock/core/`：数据抓取、标准化、指标、形态、元数据定义
- `instock/job/`：批量任务入口
- `instock/web/`：Tornado Web 服务、模板、接口处理器
- `instock/trade/`：自动交易与策略执行

辅助基础设施主要在：

- `instock/lib/`：数据库、运行模板、工具类
- `docker/`：镜像与 cron 配置
- `supervisor/`：容器内进程编排
- `cron/`：周期任务脚本

### 3.2 数据与表结构中心

`instock/core/tablestructure.py` 是全仓库最重要的元数据入口之一，集中定义：

- 各类数据库表结构
- 字段中文名与类型
- 策略绑定
- K线形态绑定
- Web 页面展示依赖的数据列信息

如果新增：

- 数据表
- Web 模块
- 指标或策略结果表

通常都要先从这里开始。

### 3.3 抓取与标准化链路

上游抓取逻辑主要在 `instock/core/crawling/`，不同来源拆成独立模块。

统一汇总和清洗在 `instock/core/stockfetch.py`，其职责包括：

- 调用不同来源抓取器
- 过滤股票范围与异常数据
- 将返回结果对齐到 `tablestructure.py` 定义的列结构
- 管理历史缓存 `instock/cache/hist/`

共享网络访问能力由 `instock/core/eastmoney_fetcher.py` 提供，负责：

- Eastmoney 请求会话
- Cookie 注入
- 代理支持
- 请求重试

### 3.4 批处理任务层

`instock/job/` 下每个文件通常就是一个独立任务入口。

`instock/lib/run_template.py` 提供统一批量参数能力：

- 当前交易日
- 单日
- 多日枚举
- 日期区间

`instock/job/execute_daily_job.py` 是主编排入口，但要注意当前源码状态与 README 描述并不完全一致。

当前实际启用的步骤包括：

- 初始化数据库
- 基础实时数据
- 综合选股
- 其它基础数据
- 收盘后数据

而指标、K线形态、策略、回测等步骤虽然已导入，但在当前主流程中处于注释状态。

### 3.5 数据库存储方式

数据库访问入口为 `instock/lib/database.py`。

特点：

- 默认数据库为 MySQL / MariaDB
- 连接参数可由环境变量覆盖：`db_host`、`db_user`、`db_password`、`db_database`、`db_port`
- 使用 pandas `to_sql` 落表
- 表创建后再补主键和索引

这意味着很多批处理任务的数据流是：

1. 抓取
2. DataFrame 清洗/对齐
3. 写入数据库
4. 补齐主键与索引

### 3.6 Web 层结构

该仓库前端不是独立 SPA，而是 Tornado 服务端渲染页面。

关键入口：

- `instock/web/web_service.py`

主要路由职责：

- 首页
- 通用表格数据页
- 指标详情页
- 自选关注写入接口

其中：

- `instock/web/dataTableHandler.py` 提供通用表格页面和表格数据接口
- `instock/web/dataIndicatorsHandler.py` 提供个股/ETF 指标详情页与关注接口
- `instock/core/singleton_stock_web_module_data.py` 根据 `tablestructure.py` 动态构造左侧菜单与模块定义

因此，UI 展示与数据库元数据是强耦合的。

### 3.7 交易模块

自动交易入口为：

- `instock/trade/trade_service.py`

特点：

- 基于 `easytrader`
- 使用 `instock/config/trade_client.json` 提供客户端配置
- `MainEngine` 负责策略装载和执行
- 当前 README 也明确该能力偏 Windows 使用场景

## 四、容器化与调度方式

容器默认并不是只跑一个 Python 进程，而是通过 `supervisor` 管理多进程。

关键文件：`supervisor/supervisord.conf`

默认拉起：

- `instock/bin/run_job.sh`
- `instock/bin/run_web.sh`
- `instock/bin/run_cron.sh`

cron 调度写死在 `docker/Dockerfile` 中，主要规则为：

- 工作日交易时段每 30 分钟跑一次基础抓取
- 工作日 17:30 跑日终流程
- 周三/周六 10:30 跑月度清理任务

这是典型的：

- Docker 负责运行环境
- supervisor 负责进程编排
- cron 负责周期触发
- Python job 负责业务执行

## 七、初始化结论

该仓库已经具备清晰的“抓取 → 清洗 → 入库 → 页面展示 / 交易使用”主链路。

当前最重要的几个事实是：

- `tablestructure.py` 是元数据中心
- `stockfetch.py` 是数据标准化中心
- `execute_daily_job.py` 是日常作业总入口，但当前启用步骤少于 README 描述
- Web 层是 Tornado 服务端渲染，不是前后端分离项目
- Docker 部署依赖 supervisor + cron 的组合，而不是单进程容器
