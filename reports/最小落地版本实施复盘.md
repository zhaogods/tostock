# 最小落地版本实施复盘

> 生成日期：2026-06-01
>
> 对照文档：`reports/项目完善路线规划.md`
>
> 范围：本报告梳理本次“最小落地版本”已完成改动、验证结果，以及与上一份路线规划中最终目标形态的差距。

## 1. 本次最小落地版本目标

本次开发不是一次完整产品化重构，而是围绕“能稳定跑、能观测、能校验、能产出结果”的最小闭环进行落地。

核心目标包括：

1. 统一配置入口，减少 `.env`、JSON、环境变量读取逻辑分散的问题。
2. 增加任务运行日志，让每日任务的成功、失败、耗时、写入量可追踪。
3. 增加非阻塞数据质量校验，让入库数据至少有基础质量记录。
4. 增加策略回测排行榜，把已有策略表中的 `rate_*` 字段汇总成可展示结果。
5. 增加每日规则化复盘报告，先生成 Markdown，不接外部 AI API。
6. 将新增能力接入主任务流程和 Web 菜单。
7. 用项目根目录 venv 执行 Python-only 验证。

## 2. 本次完成的主要改动

### 2.1 统一配置入口

新增：

- `instock/lib/config.py`

完成内容：

- 统一读取项目根目录 `.env`。
- 提供项目根目录定位：`project_root()`。
- 提供数据库配置读取：`get_db_config()`。

- 提供 Web 配置读取：`get_web_config()`。

- 支持数据库密码脱敏输出。

对应修改：

- `instock/lib/database.py`
- `instock/core/tushare_provider.py`
- `instock/web/web_service.py`

收益：

- 配置读取路径更统一。
- 数据库、Tushare、Web 配置不再各自重复解析 `.env`。
- 后续新增配置项有统一入口。

### 2.2 数据库边界与 SQL 安全改进

修改：

- `instock/lib/database.py`

完成内容：

- 数据库连接配置改为来自 `instock.lib.config`。
- 日志中的数据库连接信息改为密码脱敏。
- `update_db_from_df()` 改为参数化 SQL。
- `checkTableIsExist()` 改为参数化查询 `information_schema.tables`。
- `executeSql()`、`executeSqlFetch()`、`executeSqlCount()` 在连接失败时直接返回，减少二次异常噪声。
- `insert_db_from_df()` 默认对普通业务表执行非阻塞数据质量校验。

收益：

- 降低 SQL 拼接风险。
- 数据库不可用时错误更清晰。
- 入库路径具备基础质量观测能力。

### 2.3 新增观测与质量表

- `instock/core/tablestructure.py`

1. `job_run_log`

   - 记录每日复盘报告标题、摘要、Markdown 路径、创建时间。

收益：

- 数据库初始化流程会自动创建新增表。

### 2.4 任务运行日志

## 4. 与最终目标形态的差距

上一份报告中定义的最终目标是：

> A 股个人量化研究 + 数据看板 + AI 复盘助手

理想数据流是：

```text
每日自动更新数据
    ↓
生成行情 / 资金 / 指标 / 策略结果
    ↓
自动回测验证策略质量
    ↓
Web 展示策略排行榜和股票详情
    ↓
AI 生成每日复盘和个股分析
    ↓
推送观察池和风险提醒
```

本次最小版本相当于完成了其中的前半段骨架：

```text
每日自动更新数据
    ↓
生成行情 / 资金 / 指标 / 策略结果
    ↓
轻量汇总已有 rate_* 字段
    ↓
Web 展示策略排行榜、任务日志、质量日志、日报摘要
    ↓
规则化 Markdown 每日复盘
```

距离最终目标还差以下关键能力：

### 4.1 从“已有字段汇总”到“完整回测系统”

当前策略排行只是读取既有策略表里的 `rate_*` 字段。

最终版本需要：

- 标准化回测引擎。
- 组合级收益曲线。
- 最大回撤、夏普、年化收益、波动率等指标。
- 交易成本、滑点、成交约束。
- 策略横向对比和时间维度稳定性分析。

### 4.2 从“通用质量检查”到“可信数据治理”

当前质量检查偏通用，只能发现明显问题。

最终版本需要：

- 每张核心表的专属质量规则。
- 多数据源交叉验证。
- 缓存版本管理。
- 异常数据隔离。
- 失败重试和告警。

### 4.3 从“脚本编排”到“任务平台”

当前仍是脚本级编排，只是增加了日志记录。

最终版本需要：

- 任务依赖关系显式化。
- 任务失败重试。
- 任务状态 Web 总览。
- 可配置启停任务阶段。
- 定时任务和手动补跑统一管理。

### 4.4 从“规则化报告”到“AI 复盘助手”

当前日报是规则模板，不具备真正的自然语言推理。

最终版本需要：

- 接入 LLM。
- 结构化 prompt 和上下文组装。
- 每日市场风格判断。
- 个股风险分析。
- 策略命中解释。
- 观察池和风险名单生成。

### 4.5 从“Web 表格展示”到“研究工作台”

当前 Web 仍主要是通用表格页。

最终版本需要：

- 策略排行榜专属页面。
- 策略详情和历史表现曲线。
- 任务运行状态看板。
- 数据质量趋势看板。
- 报告在线查看和检索。
- 个股分析页与复盘报告联动。

## 5. 当前版本的价值判断

本次最小落地版本的价值不在于“已经做成最终形态”，而在于把项目从原来的数据抓取与展示系统，推进到了一个可持续演进的分析闭环：

```text
配置统一
  ↓
任务可观测
  ↓
数据可校验
  ↓
策略可横向比较
  ↓
日报可自动产出
  ↓
Web 可查看结果
```

这使后续 P2、P3、P4 的工作有了明确落点：

- 回测指标可以继续扩展到 `cn_stock_strategy_backtest_rank`。
- 更严格的数据质量规则可以继续写入 `data_quality_log`。
- 任务告警可以基于 `job_run_log`。
- AI 复盘可以替换或增强 `daily_recap.py`。
- Web 页面可以在已有菜单入口上继续增强。

## 7. 风险与注意事项

1. 当前本地数据库未完成有效连接验证。
   - 已验证数据库不可用时任务监控不阻断主任务。
   - 但真实入库、建表、Web 查询仍需要数据库环境验证。

2. 当前日报是规则化报告。
   - 不应把它理解为真正 AI 复盘。
   - 它是后续 AI 接入前的最小可用产物。

3. 当前策略排行榜依赖既有 `rate_*` 字段。
   - 如果策略表没有回测字段或当天无样本，会跳过或生成空排行。
   - 这不是完整回测系统。

4. 当前数据质量校验是非阻塞。
   - 它优先记录问题，不会阻止异常数据入库。
   - 后续可按表逐步提高严格程度。

5. `docker/cron/` 和 `docker/tostock/` 是此前 Docker 构建上下文副产物。
   - 它们不是本次最小落地版本的业务代码。
   - 提交前应确认是否需要保留。

## 8. 总结

本次最小落地版本完成的是“项目增强地基”：配置、监控、质量、排行、日报和 Web 入口。

它已经覆盖上一份路线规划中 P0/P1 的一部分，并在 P3/P4 上做了最小可用雏形；但距离最终的“A 股个人量化研究 + 数据看板 + AI 复盘助手”仍有明显差距，尤其是完整回测系统、策略平台化、AI 复盘、告警推送和专属 Web 看板仍需继续迭代。
