# 运行监控、策略菜单与每日复盘开发方案

> 生成日期：2026-06-01
>
> 范围：针对 Web 菜单分组、运行监控状态与默认日期、每日规则化复盘正文展示三个问题的开发实施方案。

## 1. 背景与目标

本次开发目标不是重构整个 Web 或任务系统，而是在现有 Tornado + 通用表格页架构上做最小闭环修复：

1. 恢复“股票策略数据”为单一菜单分组，避免“基本面选股”与其他策略项被拆开。
2. 修复运行监控：默认打开今日任务表，任务结束后正确显示 `success` / `failed`，不再全部停留在 `running`。
3. 让每日规则化复盘报告不仅写入摘要表，还能通过独立菜单分组直接查看 Markdown 正文。

## 2. 问题一：股票策略菜单分组

### 根因

左侧菜单模板按 `leftMenu.leftMenuList` 的连续 `type` 分组。当前 `TABLE_CN_STOCK_SPOT_BUY` 先出现，之后插入了“策略分析”和“运行监控”，再追加 `TABLE_CN_STOCK_STRATEGIES`，导致同为“股票策略数据”的菜单被拆成两个块。

### 方案

修改 `instock/core/singleton_stock_web_module_data.py`：

- 将 `TABLE_CN_STOCK_SPOT_BUY` 与 `TABLE_CN_STOCK_STRATEGIES` 连续追加。
- 顺序建议为：

```text
股票策略数据
  - 基本面选股
  - 放量上涨
  - 均线多头
  - ...
策略分析
  - 策略回测排行
每日复盘
  - 每日复盘报告
运行监控
  - 任务运行日志
  - 数据质量日志
```

不改数据库表结构。

## 3. 问题二：运行监控默认日期与状态

### 默认日期根因

`instock/web/dataTableHandler.py` 当前对非实时表默认使用上一交易日。`job_run_log` / `data_quality_log` 使用 `run_date`，应默认任务运行当天。

### 默认日期方案

在 `GetStockHtmlHandler.get()` 中识别运行监控分组：

- `web_module_data.type == "运行监控"` 时使用 `datetime.date.today()`。
- 其他实时表/非实时表维持原逻辑。

### 状态根因

`instock/lib/job_monitor.py` 现在：

1. 开始时 INSERT `running`。
2. 成功/失败时再次 INSERT `success` / `failed`。

但 `job_run_log` 主键为 `(run_date, job_name, start_time)`，MySQL `DATETIME` 默认秒级精度，第二次 INSERT 与第一条冲突，导致结束态无法写入。

### 状态方案

保留开始时 INSERT `running`，任务结束后 UPDATE 同一条记录：

```sql
UPDATE job_run_log
SET end_time = %s,
    status = %s,
    duration_seconds = %s,
    rows_written = %s,
    message = %s
WHERE run_date = %s
  AND job_name = %s
  AND start_time = %s
```

这样每次任务运行只保留一条日志，状态会从 `running` 正确变更为 `success` 或 `failed`。

## 4. 问题三：每日复盘正文展示

### 当前状态

已有：

- `instock/core/report/daily_recap.py`：生成 Markdown 并写入 `daily_market_report`。
- `instock/job/daily_report_job.py`：日报 job。
- `execute_daily_job.py`：主流程接入日报 job。
- `daily_market_report` 表：保存日期、标题、摘要、报告路径。

缺口：

- 当前库中没有日报记录。
- `reports/daily/**` 当前无生成文件。
- Web 只有摘要表入口，没有直接展示 Markdown 正文的页面。

### 方案

新增日报正文 Web 页面：

- 新增 handler：`instock/web/dailyReportHandler.py`
- 新增模板：`instock/web/templates/daily_report.html`
- 新增路由：`/instock/report/daily`
- 新增菜单分组：`每日复盘 -> 每日复盘报告`

页面行为：

1. 如果 URL 带 `date=YYYY-MM-DD`，展示对应日期日报。
2. 如果不带日期，优先展示今天的日报。
3. 今天没有日报时，回退展示最近一篇日报。
4. 读取 `daily_market_report.report_path` 并展示 Markdown 正文。
5. 路径必须限制在项目 `reports/daily/` 下，避免任意文件读取。
6. 初版直接用 `<pre>` 展示 Markdown，后续可再引入 Markdown 渲染。

## 5. 实施顺序

1. 修复菜单顺序与新增日报菜单分组。
2. 新增日报正文 handler、模板、路由。
3. 修复运行监控默认日期。
4. 修复 job_monitor 状态 UPDATE。
5. 运行验证：
   - Python 编译检查。
   - Web 菜单顺序脚本检查。
   - `job_monitor.run_job()` 状态写入烟测。
   - `daily_report_job` 生成日报并落表烟测。
   - Web 接口/页面响应检查。

## 6. 风险与边界

- 不清理历史 `running` 脏数据；历史数据修复应单独处理。
- 不引入 Markdown 新依赖；正文先以原始 Markdown 展示。
- 不改变日报生成规则内容，只打通生成、落表和展示链路。
- 不重构左侧菜单模板，只通过调整数据列表顺序和新增菜单项实现目标。
