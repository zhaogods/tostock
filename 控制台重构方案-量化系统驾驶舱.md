# Tostock 控制台重构方案：量化系统驾驶舱

**日期**: 2026-06-07  
**目标**: 将控制台从"任务启停器"改造为"量化系统驾驶舱"

## 一、当前系统的核心问题

### 1.1 数据层混乱

**现状问题：**

数据源分散：
- Tushare (个股日线、资金流、财务指标)
- Eastmoney 爬虫 (综合选股、龙虎榜、大宗交易)
- AkShare (ETF、部分事件)
- 本地缓存 (instock/cache/hist/)

**问题清单**：
- ❌ 不知道今天哪些数据已更新、哪些失败
- ❌ 数据质量问题发现滞后（依赖 data_quality_log）
- ❌ 数据源之间的依赖关系不清晰
- ❌ 限流配置分散（每个API独立配置）

**核心症结**：缺少**数据资产清单**和**数据血缘关系**

### 1.2 回测流程不透明

**现状问题：**

策略执行流程（从代码推断）：
1. basic_data_daily_job → 拉取行情
2. indicators_data_daily_job → 计算指标
3. strategy_data_daily_job → 10个策略筛选
4. backtest_data_daily_job → 填充未来收益率
5. backtest_rank_daily_job → 汇总排行

**问题清单**：
- ❌ 看不到某个策略当前的回测表现
- ❌ 无法追踪策略参数变化历史
- ❌ 回测结果和策略代码分离
- ❌ 无法快速验证"如果改这个参数会怎样"

**核心症结**：缺少**策略生命周期管理**和**回测结果看板**

### 1.3 控制台功能割裂

**现状**：控制台只是"任务面板"，无法回答：
- ❌ "今天的数据完整度如何？"
- ❌ "海龟策略最近7天表现怎么样？"
- ❌ "为什么今天选出的股票比昨天少？"
- ❌ "数据从拉取到回测完成花了多久？"

### 2.3 架构重构要点

#### A. 数据资产管理层

**新增组件**：
- `instock/lib/data_asset_manager.py` - 数据资产定义和状态管理
- `instock/lib/data_asset_registry.py` - 数据资产注册表

**核心类**：
```python
class DataAsset:
    """数据资产定义"""
    name: str           # 'stock_daily', 'moneyflow', 'fina_indicator'
    source: str         # 'tushare', 'eastmoney', 'akshare'
    table: str          # 数据库表名
    update_frequency: str  # 'daily', 'realtime', 'quarterly'
    expected_count: int    # 预期记录数
    quality_checks: list   # 数据质量检查规则
    depends_on: list       # 依赖的其他数据资产
    
class DataAssetStatus:
    """数据资产状态"""
    asset_name: str
    last_update: datetime
    record_count: int
    completeness: float   # 完整度 0-1
    quality_score: float  # 质量分 0-100
    issues: list          # 数据质量问题列表
```

**控制台展示**：
- 数据地图：所有数据资产的状态一览
- 血缘图：点击某个数据看它依赖哪些数据、被哪些策略使用
- 健康度趋势：近7天的数据质量变化

#### B. 策略生命周期管理

**新增组件**：
- `instock/lib/strategy_manager.py` - 策略档案管理
- `instock/lib/strategy_analytics.py` - 策略表现分析

**核心类**：
```python
class StrategyProfile:
    """策略档案"""
    strategy_key: str
    name: str
    description: str
    parameters: dict     # 当前参数配置
    version: str         # 版本号
    table_name: str      # 策略结果表
    
class StrategyPerformance:
    """策略表现"""
    strategy_key: str
    backtest_date_range: tuple
    metrics: dict        # win_rate, avg_return, max_drawdown, sharpe_ratio
    sample_stocks: list  # 最近选出的股票样例
    trend: str           # 'up', 'down', 'flat'
```

**控制台展示**：
- 策略看板：每个策略的实时表现卡片
- 策略对比：横向对比多个策略
- 策略详情页：参数历史、回测趋势图、选股样例

#### C. 执行流水线可视化

**新增组件**：
- `instock/lib/pipeline_tracker.py` - 流水线执行追踪

**核心类**：
```python
class PipelineStage:
    """流水线阶段"""
    stage_name: str      # 'data_fetch', 'indicator_calc', 'strategy_select', 'backtest'
    status: str          # 'pending', 'running', 'success', 'failed'
    start_time: datetime
    end_time: datetime
    duration: float
    dependencies: list   # 依赖的前置阶段
    task_keys: list      # 包含的任务key列表
```

**控制台展示**：
- 时间轴：从数据拉取到回测完成的完整时间线
- 依赖图：DAG 可视化，看到哪个环节阻塞了流程
- 性能分析：哪个阶段最耗时

## 三、具体实施方案

## 四、技术实现细节

### 4.1 数据库表扩展

**新增表**：

```sql
-- 数据资产状态快照（可选，也可以实时计算）
CREATE TABLE data_asset_snapshot (
    snapshot_date DATE NOT NULL,
    asset_name VARCHAR(100) NOT NULL,
    record_count BIGINT,
    completeness FLOAT,
    quality_score FLOAT,
    issues TEXT,
    updated_at DATETIME,
    PRIMARY KEY (snapshot_date, asset_name)
);

-- 流水线执行记录（阶段3需要）
CREATE TABLE system_pipeline_run (
    pipeline_id VARCHAR(64) PRIMARY KEY,
    run_date DATE NOT NULL,
    stage_name VARCHAR(50) NOT NULL,
    status VARCHAR(20),
    start_time DATETIME,
    end_time DATETIME,
    duration_seconds FLOAT,
    task_keys TEXT,
    created_at DATETIME,
    INDEX idx_date_stage (run_date, stage_name)
);

-- 策略参数历史（阶段2扩展）
CREATE TABLE strategy_param_history (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    strategy_key VARCHAR(100),
    parameters JSON,
    version VARCHAR(50),
    effective_date DATE,
    created_at DATETIME
);
```

## 五、风险与应对

### 5.1 数据库查询性能

**风险**：数据资产状态查询可能慢（每次COUNT大表）

**应对**：
- 使用缓存（Redis或内存缓存5分钟）
- 数据库添加索引（date列）
- 可选：定时任务预计算快照到 `data_asset_snapshot` 表

### 5.2 前端复杂度控制

**风险**：功能越加越多，前端代码难维护

**应对**：
- 组件化：每个面板独立 Alpine.js 组件
- 渐进式：先做核心功能，可视化图表后续加
- 文档化：关键交互写注释

### 5.3 向后兼容

**风险**：改动影响现有任务调度

**应对**：
- 只新增代码，不修改现有 task_runner/task_scheduler
- 数据资产管理层独立，不侵入原有逻辑
- 灰度发布：先上数据面板，再逐步加功能

## 七、总结

### 核心价值

1. **提升效率**：数据问题排查时间减少 80%
2. **增强信心**：实时掌握系统运行状态
3. **辅助决策**：策略表现清晰可见
4. **减少焦虑**：不再担心"是不是哪里出问题了"

### 改造原则

- ✅ 以量化流程为中心，不是任务为中心
- ✅ 信息密度高，减少点击次数
- ✅ 渐进式改造，不推倒重来
- ✅ 轻量级技术栈，避免过度工程

```
Week 1: 数据资产清单 + 健康度面板  ← 优先
Week 2: 策略表现看板
```
